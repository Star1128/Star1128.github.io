计算机网络

网络模型

也称协议栈（像栈一样层层堆叠）

OSI：国际标准化组织

五层模型与七层模型：

==应用层==

只专注于为用户提供应用功能，不用去关心数据如何传输

==传输层==

为应用层提供网络支持

==网络层==

负责实际的传输功能，寻址+路由去找到目标IP

==数据链路层==

标识网络中的设备，将目标IP->MAC地址找到具体的设备

==物理层==

将数据包转换为电信号

  • 还有一种TCP四层模型，将数据链路层和物理层合并为网络接口层

基础知识

子网掩码

分割网络号和主机号

端口号

一台主机上有很多需要接收网络数据的软件，如何知道网络包发送给哪个软件呢？

  1. 端口的范围从0~65535（2的16次方减1）
  2. 系统保留端口为：0~1024
  3. 一些常用的端口：FTP：21，SMTP：25，HTTP：80

DNS

域名系统（Domain Name System），运行在UDP上

层次结构

工作流程

URI

URI（统一资源标识符）用字符串标识互联网资源

URL（统一资源定位符）表示互联网资源的地点

URN（统一资源名称）

Socket

原理

  1. 调用socket函数创建套接字：

    int socket(int domain, int type, int protocol) 成功返回文件描述符，失败返回-1

​ domain：套接字中使用的协议簇信息（PF_INET ipv4 PF_INET6 ipv6）还是主机地址？？

​ type：套接字数据传输类型信息（TCP套接字SOCK_STREAM；UDP套接字SOCK_DGRAM）

​ protocol：计算机间通信使用的协议信息（IPPROTO_TCP；IPPROTO_UDP）

  1. 调用bind函数绑定IP地址和端口号：

    int bind(int sockfd, struct sockaddr *myaddr, socklen_t addrlen) 成功返回0，失败返回-1

​ myaddr：多播IP地址

​ addrlen：端口号，区别应用程序0-65535

  1. 调用listen函数转为可接收请求状态：

    int listen(int sockfd, int backlog) 成功返回0，失败返回-1

​ backlog：连接请求队列的最大长度

  1. 调用accept函数受理连接请求：

    int accept(int sockfd, struct sockaddr *myaddr,socklen_t *addrlen) 成功返回文件描述符，失败返回-1

  2. read()/write()：数据交换

  3. close()：关闭连接

详解connect()与accept()

TCP客户端调用connect()即发起三次握手，握手完成后Socket进入ACCEPT队列，TCP服务端调用accept()即从ACCEPT队列中取Socket

在Java中使用Socket

IP

作用

寻址（导航）+路由（操作方向盘）

分类

A类（政府）B类（企业）C类（个人）

当将IP地址写成二进制形式时，A类地址的第一位总是0，B类地址的前两位总是10，C类地址的前三位总是110（可算出地址范围）

  • 各类地址的第一个和最后一个是不可以使用的

TCP/UDP

Q：为什么需要TCP？

因为IP层是不可靠的，它不保证网络包的交付，不保证网络包的顺序，也不保证网络包中的数据的完整性。

三次握手

序号是自己上一次序号 + 上一次发送的数据包长度len

确认应答号是对方的序号 + 对方的数据包长度len

==SYN报文==是用来初始化和建立连接的，len=1

==ACK报文==表示发来的数据已确认接收无误（建立连接时的SYN报文是两端间第一次发送的消息，所以不会带ACK），len=0

#### 三次握手的目的

保证双方都有发送和接收的能力

三次握手的优点

  1. 三次握手的首要原因是为了防止旧的重复连接初始化造成混乱（防止历史连接初始化了连接）
  2. 同步双方初始序列号（确认应答号-1就可以知道对方接收到了自己的哪一条报文）
  3. 避免资源浪费

Q:两次握手可不可以？

不可以。

如果两次握手就可以建立连接的话，考虑这种情况：客户端发SYN1，但是由于网络拥塞暂未成功建立连接，然后客户端超时重发SYN2建立连接，结果服务端又收到了迟来的SYN1，又会建立冗余的无效连接，造成资源浪费；而且还不能同步双方的序列号。

Q：如果最后一个ACK丢失会怎么办？

由于服务端发送SYN+ACK后进入SYN_RCVD状态，就会启动TCP重传计时器，超时时就会重新发送SYN+ACK报文，默认总共发送5次。

客户端在没有收到重传的SYN+ACK之前，发送数据给服务端，由于服务端的状态为SYN_RCVD状态，而不是ESTABLISHED状态，就会发送RST报文给客户端，客户端就知道服务端没有收到ACK报文了。

四次挥手

为什么？因为服务端可能还有未发送完的数据

客户端服务端都能主动发起关闭请求

关于TIME_WAIT状态：

  1. 主动关闭连接的一方才有TIME_WAIT状态
  2. 持续2MSL，MSL（Maximum Segment Lifetime）即报文最大生存时间
  3. 如果没有TIME_WAIT状态，假设服务端某个数据包被网络延迟，等它到的时候客户端已经CLOSE甚至该端口已被其他应用使用，会造成数据错乱
  4. 如果没有TIME_WAIT状态，假设最后的ACK中途丢失没有送达，那么服务端触发超时重传FIN也没有响应，所以服务端会一直处于LAST_ACK状态，无法再建立其他连接

重传机制

超时重传

RTT（往返时延，数据在发送方和接收方往返一次所经历的时间）

RTO（超时重传时间，略大于RTT）

发送方发完数据包后如果超过RTO时间还没收到接收方的ACK确认报文，那么会重发该包

快速重传

发送方发送一些数据包，接收方收到Seq1包会响应ACK2（确认号=序号+1=2）

如果Seq2发送途中丢失，那么接收方下一个收到的是Seq3，但是依然响应ACK2。当发送方连续收到三次ACK2，它就知道了Seq2包丢失了，于是重传Seq2包

流量控制

滑动窗口

通过滑动窗口实现TCP流量控制

滑动窗口实际上是操作系统开辟的一块缓存空间，窗口大小就是指无需等待确认应答，而继续可以发送数据的最大值

接收方会通过TCP头的Window字段告诉发送方自己窗口还剩多少

下面三张图是发送方的滑动窗口变化示例

窗口关闭问题

当没有可用窗口时，接收方会通知发送方Window=0

当又有可用窗口时，接收方会通知发送方Window=XXX，但是如果这个通知半路丢失了，双方就会陷入死锁

解决办法：窗口探测

糊涂窗口问题

接收方目前只有几个字节的剩余窗口，也会通知给发送方，而发送方会义无反顾的发送这几个字节（TCP头部就有40字节，为了传输几字节的数据，开销太大）

解决办法：

  1. 接收方窗口太小时不必通知发送方

为接收方窗口设置MIN值，当剩余窗口大小小于MIN值时当做0处理，直接通知发送方Window=0

  1. 发送方避免发送小数据

设定数据大小，不够大就先囤积

拥塞控制

流量控制是从收发两方的角度进行控制；拥塞控制是在网络传输中进行控制，避免发送中的数据太多造成网络阻塞

拥塞控制的核心是在发送方定义一个拥塞窗口，只要网络没有阻塞，窗口就会增大，当网络中出现拥塞，窗口就会减小

慢启动

慢启动的意思就是一点一点的提高发送数据包的数量

规则：发送方每收到一个ACK，拥塞窗口的大小就会+1，所以窗口大小为指数增长

拥塞避免

当窗口大小增长到慢启动门限时，改用拥塞避免算法

规则：发送方每收到一个ACK，拥塞窗口的大小就会+1/cwnd，所以窗口大小变成线性增长

拥塞发生

当网络发生拥塞产生丢包时，会触发重传机制，根据重传机制不同，有拥塞发生和快速恢复两种处理方式

重传机制为超时重传时，采用拥塞发生算法

拥塞发生时，慢启动门限降为峰值的1/2，窗口大小直接回到解放前，即重置为1

快速恢复

重传机制为快速重传时，采用快速恢复算法

拥塞发生时（也就是快速重传中Seq2丢失的时候），慢启动门限降为峰值的1/2，窗口大小也降为峰值的1/2

随后发送方收到连续的3个ACK，拥塞窗口大小+3，重传Seq2，快速恢复结束

之后继续使用拥塞避免算法

TCP与UDP的区别

HTTP

超文本传输协议

优缺点

优点：

  1. 简单、灵活、易扩展（比如HTTPS加了一层TLS）、应用广泛成熟

  2. 无状态可以减轻服务器负担

缺点：

  1. 内容明文不加密

  2. 没有用户验证，因此信息交换双方可能会遭到伪装

  3. 接收方和发送方并不会验证报文的完整性，易篡改

  4. 无状态会增加数据传输量（别考虑Cookie，为了解决此缺点才诞生了Cookie）

状态码

状态码 类别 原因短语
1XX Informational 提示信息，接收的请求正在处理
2XX Success 请求正常处理完毕
3XX Redirection 需要客户端在响应报文中Location项找到重定向地址重新访问
4XX Client Error 客户端报文有误导致服务器无法处理请求
5XX Server Error 服务器处理请求出错

请求格式

==请求头==

  • Host 要访问的主机
  • Connection 持久连接。keep-alive可以提高效率，只要任意一端没有明确提出断开连接，则保持TCP连接状态
  • Content-Length 内容长度，指的是后面请求正文的数据长度
  • Accept 希望接收到何种MIME类型的响应内容
  • Origin 这个是源（不是标准http里的内容，因为我访问用的是chrome，是chrome加的）
  • User-Agent 用户客户端相关的信息，比如说浏览器，操作系统信息等
  • Content-Type 提交的内容类型，我提及的是json,编码是utf-8
  • Referer 来源，访问入口，比如说你在搜索引擎百度里搜索xxx，进入到网站，那么这个来源就是百度了
  • Accept-Encoding 希望接收到何种编码格式的响应内容
  • Accept-Language 希望接收到何种语言的响应内容（q属性显式指定优先级，zh-CN为默认1.0，zh为0.9）
  • Cookie cookies

==请求体==

上传多文件，每个文件的字节流数据前要有–boundary，最后一个文件要以–boundary–结尾

响应格式

  • 响应行
  • 响应头
  • 空行
  • 响应体

服务器认证和代理认证

HTTP缓存

Tips：当要使用过期缓存时，缓存服务器会向源服务器发送If-None-Match请求，如果该资源虽然过期但依然有效，源服务器会返回304并重置缓存的Age，该响应没有任何响应体，从而节省带宽

CROS跨域

允许不同源之间互相访问的机制

同源策略

同源策略是指 服务器默认只处理来自相同源的请求，除非使用跨域机制

源：如果有两个URL的协议、主机、端口都相同，那么称这两个URL是同源的

访问控制

在HTTP请求中通过Origin请求头指明请求方的协议+主机+端口

在HTTP响应中可以看到以Access-Control-Allow-XXX格式的响应头，表示允许哪些域可以访问本服务器资源

CDN

内容分发网络（Content Delivery Network）

使用户不必去源站获取数据，而是从就近的CDN服务器

HTTP1.1

优点：

  1. 使用TCP长连接 Keep-alive
  2. 使用Pipeline传输，性能↑

缺点：

  1. 会产生队头阻塞，即管道中某个请求阻塞，导致后续请求等待
  2. 请求/响应头部未经压缩，比较冗长

Q：如何优化HTTP1.1？

HTTP2

优点：

  1. 基于HTTPS。安全性↑
  2. 头部压缩（客户端服务端都会建立保存了所有字段的字典，每个字段都有一个索引号，以后只发送索引号）
  3. 报文使用二进制格式。传输效率↑
  1. 多路复用。HTTP1.1的管道传输需要服务端串行化的按序处理请求，而HTTP2采用并行处理

缺点：

  1. 多个HTTP请求在复用一个TCP连接，一旦发生丢包，就会触发TCP的重传机制，这样同一个TCP连接中的所有HTTP请求都必须等待这个丢了的包被重传回来。

HTTP3

优点：

  1. 基于UDP，UDP协议是不管数据包顺序的，所以不会出现队头阻塞问题
  2. 基于UDP的QUIC协议，该协议当发现某个流丢包时，只会阻塞这个流，其他流不受影响

存储机制

HTTP是一种无状态协议（无状态指的是浏览器对于事务的处理没有记忆能力，优点是可以减轻服务器负担）

Cookie和Session是对HTTP无状态的补充和改良

Cookie

是一种将用户数据以Key-Value形式储存在客户端里的==存储机制==，客户端每次请求都会携带Cookie

缺点：

  1. 单纯的使用 cookie，需要将用户的身份信息保存在客户端，并不安全
  2. cookie 有大小限制，以及只能使用字符串类型作为 value 值

Session

是一种储存在服务器里的==存储机制==，存储结构为ConcurrentHashMap

缺点：

  1. 服务器压力大且数据丢失后果严重
  2. 服务器负载均衡之后会出现找不到Session的情况

实例

Domain：哪些主机可以接受本Cookie

Path：主机的哪一级路径可以接受本Cookie

Expires：永久Cookie的过期时间（会话Cookie在浏览器关闭后丢失，且永远不会写入磁盘，但是一些浏览器会有会话还原的功能）

Secure：通过HTTPS协议通过加密的方式发送到服务器

HttpOnly：如果为false表示允许客户端脚本访问本Cookie，可能导致Cookie被窃取，不安全！

Token

是一种储存在客户端的==验证机制==，保存用户登录信息，提供用户身份验证

组成结构

header.payload.signature

==header==

指明算法

==payload==

包含用户信息

==signature==

用于验证消息在传递过程中有没有被更改，并且对于使用私钥签名的token，服务端还可以验证JWT的发送方是否为它所授权的发送方。

使用及验证过程

相当于是加密版的Cookie~

计算机安全

摘要算法

用于验证所传输文件的完整性

MD5

数据库里存储数据的哈希值，每次校验哈希值即可，不可逆（有被撞库攻击的风险，即常用单词的哈希值是已知的）

哈希加盐

在数据中添加随机字符，再进行哈希，防止撞库攻击，同样也是不可逆的

SHA-XX

HMAC

加密方式

对称加密

加密解密使用的密钥是同样的密钥

目前最常用的对称加密算法是AES（Advanced Encryption Standard）算法

非对称加密

发送方使用接收方的公钥加密，接收方只能使用自己的私钥解密

目前最常用的非对称加密算法是RSA算法

就好比任何人都可以知道我家的邮箱地址（公钥），都可以给我寄信，但是只有我可以用我的邮箱钥匙（私钥）打开邮箱读信

认证

把自己的数字证书给别人看，证明自己的公钥是受信任的，是经过CA机构授权的！

数字证书的颁发及校验

HTTPS

HTTPS=HTTP+TLS（原SSL）

TLS四次握手

我们探讨HTTPS的握手过程，其实就是SSL/TLS的握手过程

HTTPS结合了非对称加密和对称加密：非对称加密效率较低，HTTPS使用非对称加密使得双方都获得预主密钥之后，就开始对称加密

  1. 首先会进行TCP的三次握手，握手完成后再进行TLS的握手过程
  2. Client Hello：客户端生成第1随机数，然后将TLS版本号、客户端支持的加密方式、第1随机数发送给服务器
  3. Server Hello：服务器生成第2随机数，然后将确认支持的TLS版本号、服务器选择的加密方式、第2随机数发送给客户端
  4. Server Certificate：服务器把自己的证书发送给客户端
  5. Server Hello Down：服务器通知客户端发送完毕
  6. 客户端校验服务器的证书（CA机构的公钥已事先置入到了浏览器/操作系统里）
  7. 客户端生成预主密钥
  8. Client Key Exchange：客户端把 用服务器的公钥加密后的预主密钥 发送给服务器
  9. 服务器用自己的私钥解密就可以得到预主密钥
  10. 双方各将第1随机数+第2随机数+预主密钥结合生成 会话密钥
  11. Change Ciper Spec：客户端告诉服务器，开始使用会话密钥发送消息
  12. Encrypted Handshake Message（Finished）：客户端把之前发送的数据做个摘要，用会话密钥加密后发给服务器验证

未经安全加密，易被监听窃取伪造 通过混合加密-数字证书-摘要算法保证安全
默认端口80 默认端口443

Q：HTTPS如何解决了HTTP的缺点？

  1. HTTP明文传输—>HTTPS混合加密
  2. HTTP没有用户验证—>HTTPS数字证书
  3. HTTP不校验数据完整性—>HTTPS摘要算法
