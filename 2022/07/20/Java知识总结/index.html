Java SE

关键字

this/super

  • static方法/块中不能有this和super

    因为this和super指向的是对象（表层原因）

    非static方法的局部变量表的第一个Slot将被放入指向当前对象的this指针，static方法没有（深层原因）

  • this(参数列表)/super(参数列表)只能出现在构造方法的第一行

final

  • final不可以修饰接口
  • final修饰的类不可以被继承
  • final修饰的方法不可以被重写
  • final修饰的成员变量为基本数据类型时，在赋值之后无法改变
  • final修饰的成员变量为引用数据类型时，在赋值后其指向地址无法改变，但是对象内容还是可以改变的
  • final修饰的成员变量在赋值时可以有三种方式 (1.在声明时直接赋值 2.在构造器中赋值 3.在构造代码块中进行赋值)
  • final修饰的类中可以没有final方法
  • final变量在编译时就分配了内存

Q：为什么局部内部类和匿名内部类只能访问final变量？

为了保证内部类与外部类的数据一致性。

内外部类生命周期是一样的，内部类不会因为定义在方法中就会随着方法的执行完毕就被销毁。当外部类的方法结束时，局部变量就会被销毁了，但是内部类对象可能还存在（只有没有人再引用它时，才会死亡）。这里就出现了一个矛盾：内部类对象访问了一个不存在的变量。为了解决这个问题，就将局部变量复制了一份作为内部类的成员变量，这样当局部变量销毁后，内部类仍可以访问它，实际访问的是局部变量的”copy”。为了保证局部变量与其在内部类中的副本的一致性，就将局部变量设置为final啦~

流的分类

按照流是否直接与特定的地方（如磁盘、内存、设备等）相连，分为节点流和处理流两类。

  • 节点流：可以从或向一个特定的地方（节点）读写数据。如FileReader，FileInputStream

  • 处理流：是对一个已存在的流的连接和封装，通过所封装的流的功能调用实现数据读写。如InputStreamReader，BufferedReader。处理流的构造方法总是要带一个其他的流对象做参数

线程安全：内存可见性 + 操作原子性

volatile

  • 只能修饰==变量==

  • 无锁思想，多线程同时访问不会阻塞，==线程不安全==

  • 只保证此变量对所有的线程的可见性

    原理：

    每个线程会保留一份共享变量的副本在线程本地内存中。

    使用volatile修饰共享变量后，每个线程要操作变量时会从主内存中将变量拷贝到本地内存作为副本，当线程操作变量副本并立即写回主内存后，会通过CPU总线嗅探机制告知其他线程该变量副本已经失效，需要重新从主内存中读取。（上述过程不是原子性的）

    volatile保证了不同线程对共享变量操作的可见性，也就是说一个线程修改了volatile修饰的变量，修改后的变量会直接写回主内存，随后其他线程能立即获得最新值。

    非volatile变量在本地内存中被修改后，什么时候同步到主内存是不确定的。

  • 禁止指令重排序（指令重排序指CPU采用了允许将多条指令不按程序规定的顺序分开发送给各相应电路单元处理）

    原理：

    通过禁止指令重排序保证了数据的一致性

    有volatile修饰的变量，赋值后多执行了一个“load addl $0x0, (%esp)”操作，这个操作相当于一个内存屏障（指令重排序时不能把后面的指令重排序到内存屏障之前的位置），只有一个CPU访问内存时，并不需要内存屏障

  • volatile的读性能消耗与普通变量几乎相同，但是写操作稍慢，因为它需要在本地代码中插入许多内存屏障指令来保证处理器不发生乱序执行

ThreadLocal

ThreadLocal是无锁思想的实现

对于多线程资源共享的问题，同步机制采用了“以时间换空间”的方式，而ThreadLocal采用了“以空间换时间”的方式。

同步机制仅提供一份变量，让不同的线程排队访问；而ThreadLocal为每一个线程都提供了一份变量，因此可以同时访问而互不影响。

锁

是否需要加:lock:

悲观锁

认为自己在持有资源的时候总会有其他线程来修改资源，所以在读写时都会对资源上锁，其它线程想要请求这个资源的时候就会阻塞，直到悲观锁把资源释放为止。

Synchronized和ReentrantLock都是悲观锁思想的实现。

适用于写比较多的（线程冲突多）的情景，因为写比较少的情况下使用悲观锁会频繁进行线程的阻塞和唤醒，浪费CPU资源。

乐观锁

读时不会对资源上锁，写时通过CAS机制保证数据的一致性

适用于读多写少（线程冲突少）的情景，因为写比较多的情况下使用乐观锁会有大量线程在自旋。

:lock:的状态

对象头的运行时元数据（Mark Word）部分如下，可以看到锁分为了四种状态，不同的锁具有不同的Mark Word结构

判断一个对象是哪种锁，首先查看其锁标志位，如果是01，那么就是无锁或偏向锁，之后再向前看一位即可确定

锁只能升级不能降级

无锁

不锁定资源，线程修改不了资源就不断自旋（CAS、Volatile、ThreadLocal就是无锁思想的实现）

偏向锁

为了在只有一个线程执行同步代码块时提高性能

在锁对象的Mark Word区域的线程ID部分，记录着锁偏爱的那个线程的线程ID

当某线程想进入同步代码块的时候，判断其ID与锁记录的线程ID是否一致，一致则允许进入，不一致则升级为轻量级锁

轻量级锁

持有锁的线程的虚拟机栈中会生成Lock Record区域

线程B抢占不到锁就会自旋，当有更多的线程在自旋时，升级为重量级锁

重量级锁

同时只有一个线程能够获取锁，其余线程进入阻塞队列

CAS（Compare And Swap）算法是一种无锁算法

缺点：

  2. 长期自旋导致CPU开销大。解决办法是设置自旋次数。

  • 由于是双向链表，所以查找某个位置时，会根据位置选择从前向后查找还是从后向前查找，时间复杂度O(n/2)

Vector

==原理：数组==

  • 线程安全，里面方法都加了synchronized
  • 扩容2倍

ArrayList和LinkedList插入效率比较

在List头插入时：LinkedList快得多

在List尾插入时：十万量级LinkedList快，百万量级ArrayList快（因为数据量小的时候ArrayList会频繁扩容）

HashSet

==原理：数组+链表==

add()的原理

HashSet实际上就是HashMap，在HashSet中add(e)实际上是在HashMap中put(e,1个已经存在的空Object)….

所以add()相同元素的时候，会按照put()的原理：新key与原key的hash值和内容都相同，那么新value覆盖旧value（相同的Object对象覆盖），相当于什么也没做

由于实际上是HashMap，所以允许null值

LinkedHashSet

:fire:按添加顺序排序的HashSet，适用于需要频繁遍历

原理：存储结构不变，只是添加新元素时给新元素添加了前驱和后继，使用==双向链表==将添加进来的元素们串起来

TreeSet

:fire:按元素内容排序的Set

==原理：红黑树==

  • 只能放相同类的对象

  • 指定排序规则的方式（自定义类实现Comparable接口/给TreeSet传入Comparator参数）

负载因子0.75

使用的链地址法

当数组某一索引位置上的元素以链表形式存在的数据个数>8且当前数组长度>64时，此索引位置上的所有数据改为红黑树存储

根据hash值计算数组位置的原理

  • 高低16位异或是为了在数组length比较小的时候，也能保证高低位都可以参与到Hash的计算中，加大低位的随机性

  • Object类的hashcode()是native方法

扩容后重新计算下标的原理

扩容为2倍后实际上就是在做(n-1)&hash操作的时候，左边多参与1位，根据其为0/1来决定新下标

Hashtable

Hashtable HashMap
安全性 同步，线程安全 不同步，线程不安全
继承关系 Hashtable extends Dictionary implements Map（Dictionary类已被废弃） HashMap extends AbstractMap implements Map
是否允许空值 key和value都不允许出现null值 key和value都可以为null。因此当get()返回null值时，既可以表示HashMap中没有该键，也可以表示该键所对应的值为null。所以在HashMap中不能由get()来判断HashMap中是否存在某个键，而应该用containsKey()来判断
哈希值的使用 直接使用对象的hashCode 重新计算hash值
扩容机制 默认容量11；扩容为原容量2倍加1 默认容量16；超过最大容量*负载因子时扩容为原容量的2倍，而且一定是2的幂次倍扩容结果；而且每次扩容时，原来数组中的元素依次重新计算下标，改变存放位置并重新插入

ConcurrentHashMap

jdk1.7使用分段锁（容器里有多把锁，每把锁都只用来锁定一段数据），这样可以允许多个线程同时读写ConcurrentHashMap，在并发环境下可以实现更高的吞吐量。

而像HashTable内部是以当前HashTable对象作锁，一次只允许一个线程操作，效率较低。

jdk1.8改为CAS+Synchronized实现

WeakHashMap

常用于缓存

里面的键是弱键，弱键在下一次GC的时候被发现就会被回收，连带着键值对一起被回收
